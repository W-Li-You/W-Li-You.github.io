我的故事

我的故事

我是 李孟瑶，网名脑瓜呆呆。Java 后端开发，目前在 旺店通 供销平台方向，做搜索、中间件排查和小组协作相关的事。

下面随便写写大学到工作这几段经历。全部文章 从 2021 年 11 月排下来，和这里写的是同一段时间。

大学：在实验室里练出来的

选计算机没什么浪漫的理由，大一填志愿就是冲着就业去的。后来进了学院实验室，节奏比想象中严得多：寒假两周、暑假一周、国庆两天——差不多就这些假；没课的时间基本都在实验室，早八到晚十一点四十。

刚进去那阵子，Java 是真的跟不上。我还请过假，请到自己都不好意思了。但慢慢也适应了。大一下学期看过考研率和就业形势之后，我决定完全走就业方向，不再两头摇摆。之后开始和同学一起做项目，B 站、博客、公众号、书籍，有什么学什么，能动手就动手。

那几年打比赛、做考核项目、在力扣上补算法——不少都写在这个博客里了。算法一直不是我的长项，但练着练着也还过得去：CCPC 全国邀请赛铜奖、省赛银奖，蓝桥杯 Java 软件开发国赛一等奖（2024）。大三暑假在美团实习过一段，对后端开发有了更具体的认识。

河南科技学院 · 计算机科学与技术 · 2025 年毕业。

工作：供销平台这条线

毕业后加入旺店通。公司体量不小，但我对自己并不满意——入组考核开始就憋着劲，想尽快能独立交付。

入组头八九个月，路线大致是这样： 第一周定搜索方向并留下来 → 头三周攻坚搜索并上线 → 约一个月后带实习生做小程序 → 之后并行啃 ES 不同步、帮导师/data 组 → 五个月左右拆组当负责人 → 自己提报人才计划项目 → 六七个月转正时 P4 破格晋升 P5。

那个改变计划的第一周

入组后，我的导师在数据组，平时也很忙。他的安排是：先在供销平台待一周熟悉环境，第二周带我接手他手下另一位同事的任务——按原计划，之后我应该跟着导师去数据组。

第一周让我「先看看搜索」。后来我才慢慢明白，领导前期其实不太放心——搜索是供销平台的老问题，让一个新人待一周、了解一下就行，一周之后还是按原计划去数据组，并没指望我能改出什么名堂。我没想太多，把现有方案梳理了一遍，做了几个方向的对比，把基础方案定了下来。销售和运营侧很快看到了变化，计划于是改了：我留了下来，继续把搜索做下去。

这大概是我入职后第一个转折点——不是抢来的机会，是一周之内把事做成了。

搜索改造

头三周里，我把精力主要放在搜索业务上。

改造前，供销平台搜索的问题很具体：该出来的结果匹配不到，无关结果又多。我往下查，发现 IK 分词即使用最细粒度，对业务里的 SKU、规格字段来说还是偏粗；索引结构也和实际查询方式对不上。

这三周里重建索引、调整分词策略，搜索方案上线。销售同事后来反馈，售前演示效果好——对他们来讲，这比技术指标直观得多。

带实习生做小程序，以及 ES 不同步

小程序那边主要是接口响应慢。大约一个月后，我开始带两位实习生一起做——我给他们分工，审查代码，帮他们优化一时搞不定的接口，也教他们自己排查慢请求的方法。

带实习生这段时间之后，领导把供应链里最大的一块硬骨头分给了我：ES 不同步。这块没有「一次性改完」的方案，基本是我发现一个问题、改一个问题，慢慢压下去。客户工单几乎每天都有，失败量高峰期每天上万条，后来才一点点降下来：几千、几百，现在基本归零。

同一时期，导师那边缺人手，我也过去帮了一阵，跟着做了些 data 组的需求，接触了 Doris、DolphinScheduler 调度任务等——和啃 ES 是并行在做的，两边都在推进。

那两个月里，工作日不够，双休也用来接 ES 相关的客户反馈。最难的阶段是 ES 整体压力都很大，症状有，根因却不好定位。我是一步步加日志、梳理所有接口和访问频率、接企业微信消息通知，才慢慢把链路看清。节点 Yellow、断路器、消息体积、连接池配置……都是这么抠出来的。这些排查过程，后来整理成了博客里的 ES、MQ 系列笔记。

五个月左右，供销平台拆出独立小组，我担任小组负责人。六七个月转正时参加晋升答辩，讲了 ES 同步优化和搜索改造这两块，P4 破格晋升 P5。

带小组之后：机制、AI 与管理

组里五个人，分工大致是：我负责搜索；一人负责 ERP 转供销；一人负责价格模块；两人参与公司专项计划，其中一人经验相对少，主要在专项里帮忙。

当上小组负责人之后，我对供销平台整体业务还需要补一遍。我主动接下了组内大约一半的客户工单，从全平台的真实问题里反推业务——工单里什么模块都有，比干看文档快得多，后面推机制、对齐排期时也更有底气。

当负责人之后，我自己提报了公司人才选拔计划的项目：用 Python 做搜索无结果推荐，对照竞品看用户在搜不到的时候该怎么兜底，而不只是修「搜不到」本身。

带人之后慢慢意识到，光自己写代码不够，还得把协作和流程理顺。陆续推了一些措施——技术文档评审、工作负载看板、线上问题看板等，让供销平台的制度和规范更完整一些。组里新人多，每个人熟悉的业务面不一样；负载有时不均衡，产品排期又紧，看板用来和产品一起对齐；大 Key、慢 SQL、OOM 之类的问题频发，就想风险前置，谁发现了就记进去，可以自己选择要不要接。

工作后我也开始系统地接触 AI：自己日常开发靠 Cursor、Claude 提效；组里则把阿里开源的 Open Code Review 接进了流水线，已经全面投入使用——尽量把 AI 嵌进现有流程，而不是单独做个 demo。

管理类的书是那段时间集中补的，格鲁夫、华为相关 的读书笔记都在这个博客里。

持续在学什么

我的学习基本跟着业务走，不太单独列「学习计划」：

  • 技术深度：通读过《Elasticsearch 权威指南》，并据此写了 ES 存储引擎系列；规范类翻过阿里 Java 手册、Clean Code
  • 产品视角：《人人都是产品经理》、王慧文清华产品课——做搜索和推荐，不能只懂接口和索引，还得理解用户在找什么
  • 数据与工程：Doris、DolphinScheduler、PHP / Python，大多是因为项目需要现学现用
  • AI：Cursor / Claude 写代码和读源码；Open Code Review 做组内 Code Review 门禁

和这个博客

这个站点是笔记本，不是作品集。工作中遇到不懂的——ES、RocketMQ、OOM、大 Key 之类——会查资料、踩坑、写在这里；书读完了也顺手整理。按主题浏览可以看 **系列笔记**；想按时间翻可以从 全部文章 的 2021 年看起。

早期文章和现在的风格可能不太一样，但都留着——算是一段完整的记录。
