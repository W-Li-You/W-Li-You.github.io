《代码整洁之道》读书笔记
Published in：2025-09-10 | category： 读书笔记

整洁的代码不是某种特定的风格或模式，而是一种让代码易于理解和修改的状态。本文梳理了《代码整洁之道》一书中关于命名、函数、注释、格式、错误处理等核心知识点。

一、什么样的代码才是整洁的

整洁的代码应该具备以下特征：

  1. 可读性优先
  2. 函数短小且专注
  3. 命名揭示意图
  4. 减少阅读负担
  5. 一致的格式和结构
  6. 错误处理明确
  7. 避免重复
  8. 简单的设计
  9. 测试友好

二、如何保持代码整洁

保持代码整洁，首先要有这个意识。书中反复提到一条童子军军规：

让营地比你来时更干净。 写整洁代码需要心中始终有一个意识——我要写好我的代码，让自己和别人看着都舒服。

整洁代码的检验标准：

标准说明
新人能快速理解新团队成员能在短时间内理解代码在做什么
修改风险低修改一处代码时，不太可能意外破坏其他功能
易于调试出现问题时，能快速定位问题所在
自文档化代码本身就能说明其意图，不需要大量注释
测试友好容易为代码编写单元测试

三、命名

判断一个名称是否合格，可以问自己四个问题：

  • 意图明确吗？ 一个不了解这段代码的人能否一眼看出这个变量/函数/类是做什么的？
  • 会误导人吗？ 它有没有任何可能引起歧义的地方？
  • 简洁且可读吗？ 它是否易于发音和搜索？有没有不必要的编码？
  • 一致吗？ 它是否与项目中的其他命名风格和术语保持一致？

3.1 第一支柱：意图清晰

命名最重要的目标，是让阅读者（包括未来的你自己）一眼就能看出其意图。

① 揭示意图

名称应回答”它为什么存在、做什么事、怎么用”，必须避免模糊的、与意图无关的名称。

② 避免误导

名称必须准确，不能给阅读者提供错误的线索或暗示。

  • 避免使用专有名词，如 hp、aix、sco 不应作为变量名。
  • 慎用相似度高的名称：XYZControllerForEfficientHandlingOfStrings 与 XYZControllerForEfficientStorageOfStrings 难以区分。
  • 避免使用小写 l 和大写 O，因为它们看起来像数字 1 和 0。

③ 做有意义的区分

如果名称不同，那么其含义也必须不同，不要为了满足编译器而添加无意义的废话。

3.2 第二支柱：简洁准确

在清晰表达意图的前提下，名称应尽可能简洁、准确，并符合约定俗成的规则。

① 使用读得出来的名称

人类大脑更擅长处理词汇，而不是一堆随机的字母缩略词。

② 使用可搜索的名称

单字母名称和数字常量很难在代码库中精准定位。比起数字 7 或字母 e，常量 MAX_CLASSES_PER_STUDENT 更容易被搜索到。

③ 避免使用编码

不应将类型或作用域等信息通过前缀或特殊符号硬编码到名称里。

  • 匈牙利语标记法：在现代强类型语言中已过时，反而增加了修改变量类型时的改名成本。
  • 成员前缀（如 m_）：类和函数应该足够小，以至于不需要这些前缀来区分成员变量，现代 IDE 的语法高亮也使其变得多余。
  • 接口加 I 前缀：不推荐。更好的方式是让”实现类”有一个信息丰富的名称，接口本身用其抽象概念命名。

3.3 第三支柱：一致性

命名不是孤立的，它需要与团队、项目和环境保持一致，形成统一的语言。

① 别扮可爱

选择明确而非娱乐性的名称，明确是王道。

② 每个概念对应一个词

为每个抽象概念选择一个词，并一以贯之。先跟着项目、团队统一好用哪个词，比如项目用 save，不要又突然用 insert，更要避免在同一项目中 Controller、Manager、Driver 混用表示同一概念。

③ 别用双关语

避免同一个词用于不同目的，应”一词一义”。例如多个类中都有 add 方法表示”新增”，那么如果一个方法是”将参数与集合内的值相加”，就该命名为 insert 或 append 以避免歧义。

④ 使用解决方案领域名称

代码会被程序员阅读，所以使用计算机科学术语、算法名、模式名、数学术语是完全可以的，甚至是推荐的，例如 AccountVisitor、JobQueue。

四、函数

自查清单：

  • 首要目标：函数是否短小？能否变得更短？
  • 核心纪律：函数是否只做一件事？所有语句是否处于同一抽象层级？
  • 可读性：是否为函数起了一个描述性的、能区别不同抽象层级的名字？（长而具有描述性的名称，比长而令人费解的注释要好，比短而令人费解的名称要好得多）
  • 参数处理：函数的参数数量是否已减至最少？是否避免了标识参数？
  • 错误处理：如果函数中有 try 语句，它应该是函数的第一个单词，并且在 catch/finally 语句块后面不应再有其他内容。

4.1 第一支柱：短小

函数的物理长度是衡量其整洁度的最直观、最重要的标准。

  • 在整洁代码中，20 行长的函数已经非常罕见。
  • if、else、while 等语句中的代码块应该只有一行，通常是一个函数调用语句——这不仅能保持函数短小，也增加了函数的描述性。
  • 函数的缩进层级不应该多于一层或两层，这使得函数易于阅读和理解。

4.2 第二支柱：单一职责

只做一件事是本章最重要的规则：函数应该只做一件事，并把它做好。

“一件事”的判断标准：如果一个函数只是执行其名下同一抽象层级的操作，那么它就在只做一件事。

一个函数一个抽象层级：确保函数只做一件事的关键，是让函数中所有语句都处于同一抽象层级上。

向下规则：代码应该像一篇写得很好的文章，自上而下地阅读，每一级都向下级询问”接下来发生了什么？”。

如何重构以遵循单一职责？ 看到长函数，或函数中包含不同抽象层级的代码时，就是分解函数的时机——提取代码，放入一个名称具有描述性的新函数中。

4.3 第三支柱：优化参数

函数的参数列表定义了其与外部世界的耦合方式，更少的参数意味着更低的耦合和更易理解。

  • 更少的参数：最理想是零参数函数，其次是单参数，再次是双参数，应尽量避免三参数函数。参数越多，函数越难理解和测试。
  • 标识参数应尽量避免：向函数传入布尔值参数是一种丑陋的做法，明确表明这个函数做了不止一件事（一件是为 true，另一件是为 false）。解决方法是将此类函数拆分为两个。

4.4 第四支柱：副作用与命令查询分离

函数应该遵守承诺，不做它名称所暗示之外的事情。

① 无副作用

副作用是一种谎言：函数承诺只做一件事，但它还做了其他”隐藏”的事，比如对自己类中的变量做出未预期的改动，或把变量搞成共享/全局变量——这会导致时序性耦合和令人困惑的 Bug。

反面例子：一个名为 checkPassword 的函数，在验证密码的同时还会初始化 Session。如果在非预期时调用验证功能，就会产生副作用。

② 命令查询分离

函数要么做什么事（命令），要么回答什么事（查询），但二者不可兼得。

  • 命令（修改对象状态）：函数应修改某对象状态，但不返回任何值（void）。
  • 查询（返回数据）：函数应返回数据，但不产生副作用（不修改对象状态）。

五、注释

首要原则：代码本身是最好的文档。在写注释之前，先问自己：”我能否通过修改代码（如重命名、提取函数、引入解释性变量）来消除对这段注释的需求？”

质量优于数量：不要为了达到某种注释覆盖率而写注释，每个注释都必须有其存在的充分理由。

保持真实与精确：确保注释绝对精确，并与代码行为完全一致；一旦代码修改，立即更新或删除相关注释。

定期清理：将清理无用、过时的注释作为代码重构的一部分，把它们当作技术债务来处理。

心存敬畏：将注释视为一种”必要的恶”，仅在少数例外情况下谨慎使用。

5.1 第一支柱：注释为何是”失败”

  • 注释会撒谎：注释不像代码那样会被编译器、测试或执行环境强制检查。程序员经常修改代码却不更新注释，导致注释过时甚至与代码行为完全相反，变成精巧的、系统化的谎言。
  • 注释会贬值：代码在不断演进，而注释是静态的，没有机制保证它与代码同步更新，因此随时间推移注释的准确性必然贬值。
  • 注释制造噪音：过多无用的注释会淹没真正重要的代码和少量必要的注释，就像”狼来了”的哭喊，降低了程序员对注释的信任度。

5.2 第二支柱：注释的正确用法（必要的例外）

类型说明
法律信息版权声明、著作权声明等，有时是公司法律要求
提供信息的注释解释抽象方法返回值的具体意图，但更好的方式是改进函数/变量名来承载这些信息
对意图的解释解释某个代码决策背后的”为什么”，而非”是什么”
警示提醒其他程序员注意潜在后果或待完成的工作，如 // TODO（但不是不写代码的借口，应定期清理）
放大强调某段代码的重要性，这种重要性可能从代码本身看不出来
公共 API 文档编写良好的 Javadoc 等是必要的，但同样应避免废话

5.3 第三支柱：糟糕注释黑名单

大多数注释都属于此类，不仅无益反而有害，应像清除代码坏味道一样坚决清除或重构：

  • 喃喃自语：只对作者本人在特定时刻有意义，如 /* 这有时会出问题，不知道为啥 */
  • 冗余注释：只是用更啰嗦、更不精确的语言重复代码已经表达的内容，如 // 关闭连接 写在 connection.close(); 之上
  • 误导性注释：原本正确但因代码修改已不再准确，将阅读者引向错误方向
  • 循规蹈矩的注释：因流程要求每个函数/变量都必须有注释而编写的模板化废话，如 /** * @param name 用户名 */
  • 日志式注释：在代码开头记录每次修改的日期和人员，应由版本控制系统（Git）管理而非污染源代码
  • 废话注释：明显、无关紧要的注释，如 // 默认构造函数 写在空的默认构造函数之上
  • 可怕的标记：如 /// ---// ACTION //--- /// 这类夸张标记框，只会制造视觉混乱
  • 注释掉的代码：看到就立即删除！没人敢删除已注释的代码因为它”可能还有用”，但版本控制系统的存在就是为了帮你找回旧代码

六、格式

自查清单：这份代码的格式是否符合团队的编码规范？是否使用了自动化工具来保证一致性？

每个程序员都有自己喜欢的格式规则，但如果在团队工作，就是团队说了算——一组开发者应当认同一种格式风格，每个成员都应该采用那种规则。

6.1 第一支柱：垂直格式——营造”报纸”式体验

代码文件在垂直方向上的组织，决定了读者能否快速把握其全貌和脉络。

① 文件大小：向报纸学习

源文件应像一篇报纸文章：标题（名称）点明主题，开头（顶部）是高层概念和算法，细节在下方逐渐展开。大多数整洁的代码文件在 200–500 行 之间；长文件并非不可接受，但应特别留意其组织方式。

② 垂直密度：关联概念应垂直紧邻

紧密相关的代码行（如一个变量声明和其紧接着的使用）应在垂直方向上彼此靠近，形成一个代码块，它们之间的空行应该很少或没有。

③ 垂直距离：概念关联性决定物理距离

元素规则
变量应声明在尽可能靠近其使用位置的地方；局部变量声明在函数顶部，循环控制变量可在循环语句中声明
实例变量应声明在类的顶部
相关函数若一个函数调用了另一个，应在垂直方向上尽可能靠近；调用者尽可能放在被调用者之上，形成自然流畅的自上而下阅读顺序

④ 概念垂直排序：自上而下

函数调用依赖链应自上而下排列，被调用的函数放在执行调用的函数下面，源文件可以从顶部的主要概念开始，逐步向下阅读到最低层的细节。

6.2 第二支柱：水平格式——追求短小与清晰

代码行在水平方向上的长度和内容，应便于一眼阅读，无需水平滚动。

① 行长度：保持简短

应尽力保持代码行简短，推荐上限是 80 个字符，最多不超过 120 个字符——人类的眼睛在阅读较短文本行时更为舒适。

② 水平空格与密度：用空格强调关联与分离

  • 赋值运算符两边加空格，强调左右两边。
  • 函数名和左括号之间不加空格，强调它们是一体的。
  • 函数参数之间用空格隔开，强调它们是分离的。

③ 水平对齐：通常不必

对齐声明和赋值（如让所有等号或变量名在一条竖线上）通常没有必要，这会让读者目光停留在不对齐的行末尾，反而增加阅读负担。对齐有时反而指出了代码中的”坏味道”，比如声明了一组过长的变量。

④ 缩进：体现代码的层次结构

缩进是代码结构在视觉上的直接体现，至关重要：文件顶层的类不缩进，类内部的方法缩进一层，方法内的代码块再缩进一层，以此类推。绝对不要破坏缩进规则，即使在简短的 if 语句、循环或方法中也不应写成一行。

6.3 第三支柱：团队规则与编码规范

格式不是个人偏好，而是团队共识和纪律：

  • 每个软件开发团队都应采用一套统一的代码格式规则，理想情况下所有成员都同意并遵守它。
  • 使用现代化的 IDE 和代码格式化工具（如 IntelliJ IDEA、Eclipse、Prettier 等）来强制执行团队规则，一键格式化为标准样式，消除无谓的争论和个人风格差异。

七、错误处理

自查清单：

  • 机制选择：是否使用了异常，而非通过返回码来报告错误？
  • 代码结构：编写可能出错的代码时，是否先构思了 try-catch-finally 结构？函数是否将错误处理作为唯一职责（即 try-catch 块后无其他代码）？
  • 上下文清晰度：抛出的异常是否包含足够信息，能让调试者快速定位问题和原因？
  • 空值管理：是否避免返回 null？是否避免传递 null 参数？
  • 主体逻辑可见性：错误处理代码是否没有淹没和干扰主要业务逻辑？阅读者能否一眼看清程序的正常执行路径？

7.1 第一支柱：使用异常

在现代化编程语言中，异常提供了比错误码更优越的错误处理机制。

① 别返回错误码，抛出异常

使用异常可以将错误处理代码从主路径代码中分离出来，使代码更简洁。

② 先写 Try-Catch-Finally

异常会定义程序执行的一个范围。在编写可能抛出异常的代码时，最好先写出 try-catch-finally 语句的框架——这有助于定义代码用户应该期待什么，无论 try 块中代码成功还是失败，也迫使你首先关注错误处理，从而产生更健壮的设计。

7.2 第二支柱：提供异常上下文

① 创建信息充分的错误消息

抛出的每个异常都应提供足够的上下文信息，包括操作失败的类型和失败原因，应将这些信息打包到异常类型中，或通过构造函数传递给异常。

② 定义异常类时考虑其调用方

关键是考虑它们如何被捕获：

  • 如果在编写一个关键的第三方库，可以定义多个细粒度的异常类。
  • 但对于大多数应用程序开发，对单一 catch 块中捕获的不同错误进行归类打包通常更有用——可以定义一个基类，再为不同类型错误创建子类，并包含足够的上下文信息。

7.3 第三支柱：业务逻辑与错误处理分离

这是本章核心思想，旨在防止错误处理淹没代码的主体逻辑。

① Try-Catch 代码块的范围与结构

try 块中的代码就像事务，catch 块则保证程序维持在一种持续状态，无论 try 中发生了什么。try-catch 代码块在视觉上应该具有一定的”丑陋性”——这很好，因为它强调了错误处理的重要性，并鼓励你将其最小化。

② 错误处理就是一件事

函数应该只做一件事，而错误处理就是”一件事”。因此一个处理错误的函数不应再做其他事——这意味着在 try 或 catch/finally 代码块之后，不应该再有其他代码，确保了错误处理职责的单一性。

7.4 第四支柱：特定场景与最佳实践

① 不要返回 Null

  • 危害：返回 null 本质上是在给调用者制造麻烦，要求他们必须处理 null 的情况，导致代码中遍布防御性的 if (obj != null) 检查，掩盖了真正的逻辑。
  • 解决方案：当方法可能找不到结果时，返回一个空集合（如 Collections.emptyList()）或一个特例对象，而不是 null；或者直接抛出异常。

② 不要传递 Null

  • 危害：将 null 作为参数传递给其他方法是更糟糕的行为，迫使被调用方法必须对参数做 null 检查，否则可能在运行时抛出难以调试的 NullPointerException。
  • 解决方案：在大多数情况下，向方法传递 null 是一种逻辑错误，应通过代码设计和约定来避免，可在 API 层面通过 @NotNull 等注解或断言来明确禁止。

总结

整洁代码的核心，归根结底是对阅读者的尊重——无论是命名、函数、注释、格式还是错误处理，每一条规则背后都指向同一个目标：让代码自己说话，减少他人理解和维护这段代码的认知负担。记住那条童子军军规：让营地比你来时更干净。
